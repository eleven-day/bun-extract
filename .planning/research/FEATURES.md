# Feature Research

**Domain:** CLI automation toolkit — single binary, multi-modal input routing
**Researched:** 2026-03-22
**Confidence:** HIGH (stack well-established; routing pattern inferred from evidence)

---

## Feature Landscape

### Table Stakes (Users Expect These)

Features users assume exist. Missing these = product feels incomplete.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| stdin / stdout / stderr separation | Unix pipeline contract — any CLI that can't pipe is broken | LOW | stderr for logs/errors, stdout for machine-readable output; violating this makes the tool unchainable |
| `--help` and `--version` flags | Every CLI ships these; omitting them signals immaturity | LOW | Auto-generated by arg-parsing library (e.g., `citty` or `yargs`) |
| Config file for API keys | Secrets must not live in shell history; users expect `~/.toolname/config.toml` or env var override | LOW | Follow `XDG_CONFIG_HOME` on Linux; `~/Library/Application Support` on macOS; env vars always take precedence |
| Error messages with actionable text | "command not found" vs "API key missing — set OPENROUTER_API_KEY in config" | LOW | Human-readable on stderr; structured (JSON) on stdout when `--json` flag used |
| Web URL scraping and text extraction | Input-to-text is the core value; fetching a URL is the most common trigger | MEDIUM | Must handle JS-heavy SPAs vs static HTML; `undici` or `cheerio` for static; puppeteer/playwright only if JS rendering is required |
| File text extraction (PDF, plain text, markdown) | File processing is equally core to web scraping; PDF is the dominant document format | MEDIUM | `unpdf` (modern, TypeScript-native, cross-runtime) is the right library; handles most PDFs without native binaries |
| Single binary distribution via `bun build --compile` | Eliminates the environment setup problem that motivates this tool's existence | MEDIUM | Bun v1.2+ supports cross-compilation to Linux x64/ARM64, macOS x64/ARM64, Windows x64/ARM64 |
| Output to stdout or file (`-o / --output`) | Users need to redirect to file without shell redirection gymnastics | LOW | `--output <path>` flag; default to stdout |
| Exit codes that reflect success/failure | Scripting depends on `$?`; a tool that always exits 0 cannot be used in pipelines or CI | LOW | 0 = success, 1 = input error, 2 = auth error, 3 = network error; document codes |

### Differentiators (Competitive Advantage)

Features that set the product apart. Not required, but valued.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Auto-routing engine by input type | Zero-config: pass anything (URL, file path, GitHub URL) and the right handler fires — no subcommands to remember | MEDIUM | Detect: URL scheme (`http/https`), GitHub URL pattern (`github.com/*`), file extension (`.pdf`, `.mp4`, `.mp3`, `.png`, `.jpg`), MIME type fallback; this is the core UX differentiator |
| GitHub repo summarization | Combines crawling (README, directory structure) + LLM summarization in one command; repomix/gitingest prove demand (20k+ GitHub stars) | MEDIUM | Fetch `README.md`, `tree` structure via GitHub API (no auth for public), pass to OpenRouter LLM; respects `--depth` for large repos |
| OCR with cloud + local fallback | Privacy-sensitive users can run Tesseract/PaddleOCR locally; cloud (Google Vision, Azure) for accuracy when available | HIGH | Cloud: OpenAI Vision via OpenRouter multimodal or dedicated OCR API; local: Tesseract via shell subprocess (universally available, CPU-only); VLM-based alternatives (DeepSeek-OCR) need GPU — avoid as default local path |
| ASR with cloud + local fallback | Same dual-mode philosophy as OCR; audio transcription is expensive to call cloud every time | HIGH | Cloud: OpenAI Whisper API via OpenRouter or direct; local: `whisper.cpp` (macOS/Apple Silicon) or `faster-whisper` subprocess; local works well on M1/M2/M3 — matches author's primary target |
| Video processing pipeline (FFmpeg) | Extract audio → ASR transcription, keyframes, metadata, subtitle extraction in a single command chain | HIGH | FFmpeg as external dependency is acceptable and expected for video; `bun.spawn()` to call FFmpeg/ffprobe subprocesses; pipeline: probe metadata → extract audio → ASR → output |
| LLM integration via OpenRouter | Single API key for 500+ models; model-agnostic summarization without vendor lock-in | MEDIUM | OpenAI-compatible REST API; use `fetch()` directly (Bun has native fetch); support `--model` flag override; OpenRouter `":floor"` shortcut for cost routing |
| `--json` output mode | Enables downstream scripting: `bun-kit <url> --json \| jq '.text'` | LOW | All handlers emit a consistent `{ input, type, text, metadata }` envelope when `--json` is set |
| Bytecode compilation for faster startup | `bun build --compile --bytecode` reported 2x faster cold start — matters for CLI tools called in loops | LOW | Single flag addition at build time; no code change required |

### Anti-Features (Commonly Requested, Often Problematic)

Features that seem good but create problems.

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Plugin / extension system | "Let users add new handlers" sounds extensible | Breaks single-binary guarantee; plugin discovery requires runtime resolution; v1 scope creep with high maintenance surface | Build all handlers in-tree for v1; fork-and-PR is the extension model until usage patterns are clear |
| GUI / web interface | "More accessible for non-CLI users" | Contradicts the core value (pipe-friendly, portable binary); doubles scope; adds a web server runtime dependency | Stay CLI-only; pipe output to other tools for display |
| Real-time streaming / live transcription | "Transcribe audio as it plays" | Batch model (Whisper) is not designed for streaming; ASR streaming requires a different architecture (Deepgram, WebRTC); adds WebSocket/SSE complexity | Batch processing only in v1; revisit if a streaming handler is needed later |
| Database / history of past runs | "Remember what I processed before" | Adds SQLite dependency, migration logic, disk I/O; personal tool doesn't need this for v1 | Redirect stdout to files; use shell history for rerun; SQLite logging is a v2 feature if users request it |
| Multi-user / server mode | "Run as a shared service" | Requires auth, rate limiting, session isolation — different product entirely | Out of scope by design; run locally |
| Interactive TUI / prompts | "Guided input for first-time users" | Breaks piping; adds `inquirer`/`@clack/prompts` dependency; conflicts with `--json` mode | Self-documenting `--help`; clear error messages that tell users exactly what flag is missing |
| Auto-update mechanism | "Stay current automatically" | Binary auto-update requires signature verification, download + replace logic, admin perms on some systems | Distribute via GitHub Releases with install script; users pin or update manually |

## Feature Dependencies

```
[Single Binary Distribution]
    └──requires──> [Bun build --compile]
                       └──requires──> [No native Node.js addons in critical path]

[Auto-Routing Engine]
    └──requires──> [Input Type Detection]
    └──dispatches to──> [Web Scraper]
                     ──> [GitHub Summarizer]
                     ──> [PDF Parser]
                     ──> [OCR Handler]
                     ──> [ASR Handler]
                     ──> [Video Pipeline]

[Video Pipeline]
    └──requires──> [FFmpeg (external)]
    └──requires──> [ASR Handler] (for audio extraction → transcription)
    └──uses──> [Metadata extraction via ffprobe]

[GitHub Summarizer]
    └──requires──> [Web Scraper] (fetches raw files from github.com)
    └──requires──> [LLM Integration] (summarization step)

[OCR Handler]
    └──cloud path──> [OpenRouter / Vision LLM]
    └──local path──> [Tesseract subprocess] (no LLM needed)

[ASR Handler]
    └──cloud path──> [OpenRouter / Whisper API]
    └──local path──> [whisper.cpp / faster-whisper subprocess]

[LLM Integration]
    └──requires──> [Config file] (OPENROUTER_API_KEY)

[JSON Output Mode]
    └──enhances──> [All handlers] (unified envelope)
```

### Dependency Notes

- **Auto-Routing requires Input Type Detection first:** The detection logic (URL vs file path vs GitHub URL vs stdin) is the foundation for all handlers; it must be built before any handler is wired up.
- **Video Pipeline requires both FFmpeg and ASR Handler:** The audio extraction step outputs a temporary WAV/MP3 which is fed into the ASR handler; they share the same transcription code path.
- **GitHub Summarizer requires LLM Integration:** The crawl step can work standalone (dump raw text), but the summarization step that makes it valuable requires OpenRouter.
- **OCR and ASR cloud paths require Config:** Both depend on API keys; the tool must fail gracefully with a clear error if keys are absent, falling back to local if available.
- **Single Binary requires avoiding native Node.js addons:** Libraries like `canvas`, `sharp` (native), or `node-gyp`-based packages break `bun build --compile`; all selected libraries must be pure JS/TS or use Bun's native APIs.

## MVP Definition

### Launch With (v1)

Minimum viable product — what's needed to validate the concept.

- [ ] Input type detection and auto-routing engine — this is the core UX; without it, the tool is just a collection of subcommands
- [ ] Web scraping (static HTML) — highest-frequency use case; URL-in, text-out
- [ ] PDF text extraction — second most common file type; use `unpdf`
- [ ] Config file + env var support for API keys — required for LLM and cloud features to work
- [ ] LLM summarization via OpenRouter — ties web scraping and GitHub handler together; needed for GitHub summary feature
- [ ] GitHub repo summarization — validated demand (repomix has 20k stars); compelling demo case
- [ ] `--output`, `--json`, `--help`, `--version` flags — table stakes; without these the tool is not usable in pipelines
- [ ] Single binary build for macOS ARM64 and Linux x64 — author's machines; validates the distribution model

### Add After Validation (v1.x)

Features to add once core is working.

- [ ] OCR (cloud first, Tesseract local fallback) — add when web+PDF flow is stable; cloud path is simpler to ship first
- [ ] ASR (cloud first, whisper.cpp local fallback) — same pattern as OCR; validate cloud before local complexity
- [ ] Video pipeline (FFmpeg → audio extraction → ASR) — builds on ASR handler; add once ASR is stable
- [ ] Video keyframe and subtitle extraction — add-on to video pipeline; lower value than transcription
- [ ] `--model` flag for OpenRouter model selection — useful once LLM integration is running
- [ ] Windows x64 binary — cross-compile is a single flag change; add when macOS/Linux are stable

### Future Consideration (v2+)

Features to defer until product-market fit is established.

- [ ] VLM-based OCR (DeepSeek-OCR, PaddleOCR-VL) — requires GPU inference; complexity doesn't fit personal-tool profile; revisit if Tesseract accuracy is a recurring complaint
- [ ] Run history / SQLite logging — only add if users explicitly ask for it; adds migration overhead
- [ ] `--watch` mode for file processing — only relevant if batch processing of changing inputs becomes a use case
- [ ] Streaming ASR — different architecture entirely; requires a streaming-capable backend

## Competitor Feature Analysis

| Feature | `llm` (Simon Willison) | `repomix` | `whisper` CLI | bun-kit approach |
|---------|----------------------|-----------|--------------|-----------------|
| Input auto-routing | No — explicit subcommand per input type | No — repos only | No — audio files only | YES — core differentiator |
| LLM integration | YES — plugin system, multi-provider | No — dumps raw text | No | YES — OpenRouter, model flag |
| GitHub crawl | No | YES (primary use case) | No | YES — README + structure |
| PDF parsing | No | No | No | YES |
| OCR | No | No | No | YES — cloud + local |
| ASR | No | No | YES (primary use case) | YES — cloud + local |
| Video processing | No | No | No | YES — FFmpeg pipeline |
| Single binary | No — Python pip install | YES — npx / npm | No — Python | YES — Bun compile |
| Plugin system | YES | No | No | NO (anti-feature for v1) |
| Web scraping | No | No | No | YES |

**Key observation:** No existing tool covers all of these input types in a single binary. `bun-kit`'s auto-routing across URL / GitHub / file / video is its genuine gap in the market.
